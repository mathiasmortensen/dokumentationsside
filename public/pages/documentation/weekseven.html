Placement of Code

Placing your very specific code at the top of your HTML files can cause unexpected errors, however there is a good reason behind this..

When HTML gets parsed, it parses from top to bottom, meaning that the document will be read sequentially. 


<head>
  <script>
    console.log(document.getElementById("test"));
  </script>
</head>

<body>
  <div id="test"></div>
</body>

With this code example, the script gets loaded before the div with the test ID.

This means that the script is trying to grab an element that is not yet initialized, and because of this it will return null..

Instead you want to place the script at the bottom, so it gets executed after the initialization of the tag.

Synchronous vs Asynchronous when reading files

I would argue that using asynchronous methods is generally better for reading files, as it allows the program to continue executing other code while waiting for the file read operation to complete.

This also means that the main thread wont be blocked and the same thing goes for the user interface.

On the other hand we have synchronous methods, that block the main thread until the file read is done.

In some cases, if the files that are being read are small, one could still use synchronous without making the application suffer performance losses. 


// Synchronous
app.get("/data", (req, res) => {
const data = fs.readFileSync("smallFile.json");
res.send(data);
});

app.get("/data", async (req, res) => { 
const data = await fs.readFile("hugeFile.json", "utf-8");
res.send(data);
});

This would be my approach to handling file reading in Node.js Express.

Server Side Rendering vs Client Side Rendering

Server Side Rendering involves generating the HTML on the server before sending it to a client. However, in some cases the server can also serve static HTML files, which provides similar benefits.

Client Side Rendering on the other hand, involves sending the file to the client and then having the client render it.

Another important point is that you can make the server load files from startup, saving memory instead of loading them every time a request is made.

So how are the different approaches?


// Server Side Rendering
app.get("/", (req, res) => {
  const html = fs.readFileSync("index.html", "utf-8");
  res.send(html);
});

// Client Side Rendering
app.get("/", (req, res) => {
  res.sendFile(path.join(__dirname, "index.html"));
});

Nodemon EXT

Nodemon is a tool that helps develop node.js applications beacuse it automatically restart the application if files are changed.

Nodemon is however also very customizable.

You're able to create config files in the .json format, for example:


{
  "watch": ["src"],
  "ext": "js,json",
  "ignore": ["src/**/*.test.js"],
  "exec": "node src/index.js"
}

This means that whenever it detects changes in the src directory specifically js and json files, it will restart the application.

Routers in Express

Routers are a critical part of the Express.js framework, allowing you to route users to different parts of your application. 


pagesRouter.js (FILE)

import { Router } from "express";

const router = Router();
router.get("/", (req, res) => {
  res.send({data: "hello i am under the water"});
 });

export default router;

In this example, a simple router is created with a single route that sends a JSON response with the message "hello i am under the water".

But this router doesn't work yet, because it needs to 'used' in the main application.

Here's how its done..


app.js (FILE)
import express from 'express';

const app = express();

import pagesRouter from './routers/pagesRouter.js';

app.use(pagesRouter);

const PORT = process.env.PORT || 8080;

app.listen(PORT, (error) => {
    if (error) {
        return console.log(`error starting the server: ${error}`);
    }
    console.log(`server is running port: ${PORT}`);
});

By importing the pagesRouter from './routers/pagesRouter.js' we are now able to let the main application use the routes defined in the external router file.
